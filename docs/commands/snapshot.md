# `otf-release snapshot`

**Non-interactive, CI-oriented ephemeral release from the current git commit.**

```
otf-release snapshot
```

> **Experimental.** The command exists, but multi-adapter semantics, generated notes, rollback
> expectations, and workflow polish still need hardening. See [roadmap.md](../roadmap.md).

Implemented in `crates/core/src/snapshot.rs`.

## Why it exists

Snapshot releases publish hash-stamped prerelease versions from CI without going through the
interactive [`version`](./version.md) flow. From the changelog:

- **v0.5.0** — Snapshot setup was **removed from `init`** (no `snapshot.yml` generation during
  setup). Snapshot releases remain available through this dedicated command instead.
- **v0.5.0** — npm auth for snapshot and release workflows was unified on the `NPM_TOKEN` repo
  secret.
- **npm adapter** — Snapshot-style prereleases publish under their own dist-tag (for example
  `1.2.3-dev.<hash>` → `--tag dev`) so an automated snapshot never lands on `latest`.

## What it does

1. Read the short git hash of the current commit.
2. Discover publishable packages via the enabled adapter and compute snapshot versions:
   `{core}-{tag}.{hash}` (for example `1.2.3-snapshot.a1b2c3d`). The `{tag}` segment comes from
   `snapshot_tag` in `release.toml` (default `snapshot`), editable via [`config`](./config.md).
3. Run `pre_version` hooks, write versions, update internal dependency ranges, refresh the
   lockfile, then run `post_version` hooks.
4. Hand off to the standard [`publish`](./publish.md) flow for registry push and tagging.

Snapshot does not open a release PR, rewrite changelogs, or run the [`version`](./version.md)
preflight/selection UI.

## Limits today

| Area | Current behavior |
| --- | --- |
| Adapters | Runs through one adapter invocation per enabled ecosystem; multi-adapter contract is still being defined. |
| Matrix binaries | No matrix build stage — `require_staged` is empty, so matrix packages are not guarded the same way as a full release. |
| Changelog | No curated or generated changelog rewrite for snapshot runs. |
| Workflow | Not generated by `init`; wire CI to call `otf-release snapshot` yourself if you use it. |
