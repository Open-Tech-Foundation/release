# The single `release.yml` model

`otf-release` replaces the changesets-era setup (a separate `workflow_dispatch` plus a
`private:true` guard) with **one** workflow and a proper dependency DAG. The workflow is
generated by [`init`](./commands/init.md); this doc explains its shape and the reasoning.

## The shape

```
build-matrix  (asset packages only)  ‚îÄ‚îÄneeds‚îÄ‚îÄ‚ñ∂  publish  (ALL packages, topological order)
```

- **`build-matrix`** ‚Äî present only if any [asset packages](./README.md#glossary) exist.
  Cross-compiles the configured target triples and uploads the binaries as artifacts.
- **`publish`** ‚Äî `needs: build-matrix`; downloads artifacts into `.artifacts/`; runs
  `otf-release publish`, which releases the **entire topological set** (libraries *and* asset
  packages) in dependency order, attaching staged binaries where they exist on disk.

Trigger: a merge to `main` (i.e. merging a release PR produced by
[`version`](./commands/version.md)).

## Why one workflow

| Property | How the single model gives it |
| --- | --- |
| **All-or-nothing** | If any matrix leg fails, `publish` never runs. Intrinsic to `needs:`, not bolted on. |
| **Correct ordering** | "Asset package depends on freshly-published libs" is solved by topological publish in **one run** ‚Äî libs first, asset package after. No manual two-phase, no "run after libs released." |
| **No guard hack** | Asset packages are first-class publishable packages with a binary target. No `private:true`, no flip-off step. |
| **Stateless** | `publish` reads what to attach from `.artifacts/<pkg>/` on disk ‚Äî nothing persisted. |

## cargo (binary release) shape

`init --adapter cargo` generates a different shape ‚Äî a **binary** release, not a registry publish:

```
build-matrix  (cross-compile each target on its runner)  ‚îÄ‚îÄneeds‚îÄ‚îÄ‚ñ∂  release  (GitHub Release)
```

- **`build-matrix`** ‚Äî one leg per target triple, each on a matching runner (`runner_os` maps
  `*windows*`‚Üí`windows-latest`, `*apple*`/`*darwin*`‚Üí`macos-latest`, else `ubuntu-latest`). Builds
  `cargo build --release -p <pkg> --target <triple>` and uploads the binary as an artifact.
- **`release`** ‚Äî `needs: build-matrix`; downloads the artifacts and runs `gh release create
  vX.Y.Z` with them attached. The version is read from `[workspace.package] version`; the step is
  **idempotent** (`gh release view` skips an existing tag). **No crates.io, no `cargo publish`** ‚Äî
  the artifacts are how users install the binary per OS.

Auth: the default `GITHUB_TOKEN` with `contents: write` (to create the tag and Release). The git
tag `vX.Y.Z` is created by the Release step, on the merge commit.

## npm specifics

Secrets/auth: `NODE_AUTH_TOKEN`.

**Gotchas to keep** (see [adapters/npm.md](./adapters/npm.md)):

- Idempotent `npm view` skip ‚Üí resumable after a partial failure.
- `--no-workspaces` ‚Üí the private root workspace would otherwise skip packages.
- `--access public` ‚Üí scoped package first publish.
- Brotli compression via Node `zlib` ‚Üí no runner-side CLI dependency.

**Gotcha to drop:** the `private:true` guard flip, entirely.

## The generated file is yours

`init` emits an **editable scaffold**. Matrix triples carry a `# edit me` marker, and
repo-specific build steps are expected to be added by hand. Re-running `init` warns before
overwrite (`--force` to replace) and does not try to manage the file after generation.
